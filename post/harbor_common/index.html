3

type Client struct {
	modifiers []modifier.Modifier
	client    *http.Client
}

相对于普通的 http.client 来说，增加修改功能主要用来给发送的请求进行授权。整个 harbor 中实现了以下 5 种授权器： - SecretAuthorizer：使用指定的 secret 给请求加密，通过调用secret.AddToRequest(req, s.secret)给请求头增加一个Authorization字段，内容为Harbor-Secret secret。 - UserAgentModifier：给请求头增加 “User-Agent”，实际的执行语句为req.Header.Set(http.CanonicalHeaderKey("User-Agent"), u.UserAgent) - basicAuthCredential：给请求头中增加用户密码信息。r.Header.Set("Authorization", "Basic "+basicAuth(username, password)) - notaryAuthorizer：给请求头增加 token 信息，实际执行语句为req.Header.Add(http.CanonicalHeaderKey("Authorization"),fmt.Sprintf("Bearer %s", n.token)) - tokenAuthorizer： 给请求增加 token 信息，实际的执行语句为req.Header.Add(http.CanonicalHeaderKey("Authorization"), fmt.Sprintf("Bearer %s", token.Token))

如果创建的客户端设置了授权器，那么在发送请求之前会根据 client 中存储的授权器对request进行修改授权，将授权后的请求发送出去。其中的GET,PUT,DELETE,POST请求的发送都经过了授权器的修改。

registry

在深入介绍这个模块之前，需要介绍几个概念。

第一个是Transport，对于http客户端，可以使用不同的实现了 RoundTripper 接口的Transport实现来配置它的行为。RoundTripper 有点像 http.Client 的中间件，使用golang net/http库发送http请求，最后都是调用 transport的 RoundTrip方法，详细参考 go 目录。

####registry.go 在registry.go中主要定义了registry client 以及对 registry 的一些操作。测试 registry 是否可用的 ping 操作，获取 registry 大致信息大catelog操作

repository.go

主要用来构建操作 registry 所需要的各种 API 请求，构建完成之后直接发送给 registry 组件，对镜像进行操作

定义了镜像存储库的信息，其结构体如下：

1
2
3
4
5

type Repository struct {
	Name     string
	Endpoint *url.URL
	client   *http.Client
}

这些构建的请求都是发送给私有的 registry 的，在repository中定义了如下的方法：

  • ListTag：获取镜像仓库中的镜像的标签,构建请求的格式为%s/v2/%s/tags/list", endpoint, repoName
  • ManifestExist:检查 repository 中是否有指定镜像的manifest，构建的请求为HEAD /v2/<name>/manifests/<reference>
  • PullManifest: 获取镜像的 manifest ，pull 一个镜像分为二个过程，第一步就是找镜像的 manifest，构建的请求为GET /v2/<name>/manifests/<reference>
  • PushManifest：上传通过name和reference标识的manifest，其中reference为tag或digest，构建的请求为PUT /v2/<name>/manifests/<reference>
  • DeleteManifest：删除由名称和引用标识的manifest，manifest只能通过摘要删除。构建的请求为DELETE /v2/<name>/manifests/<reference>
  • MountBlob：从另一个存储库中装载由装载参数标识的blob，具体的用法是在给镜像retag的时候使用到，如果原镜像和目标进行不在同一个repository 中，就需要进行 Mount 操作。构建的请求为POST /v2/<name>/blobs/uploads/?mount=<digest>&from=<repository name>
  • DeleteTag：调用DeleteManifest来删除
  • BlobExist：检查 repository 中是否存在指定的 Blob，构建的请求为HEAD /v2/<name>/blobs/<digest>
  • PullBlob：从 digest的registry中检索blob，构建的请求为GET /v2/<name>/blobs/<digest>
  • initiateBlobUpload： 启动可恢复的blob上传，构建的请求为POST /v2/<name>/blobs/uploads/
  • monolithicBlobUpload：完成指定 uuid 的镜像上传，构建的请求为PUT /v2/<name>/blobs/uploads/<uuid>?digest=<digest>
  • PushBlob：先调用initiateBlobUpload创建目录，然后调用monolithicBlobUpload上传指定uuid的blob。
  • DeleteBlob：删除指定 repository和digest标识的blob， 构建的请求为DELETE /v2/<name>/blobs/<digest>

关于上述请求的详解请参考：HTTP API V2 接口详细解析

func (t *Transport) RoundTrip(req *http.Request) (*http.Response, error) {
	for _, modifier := range t.modifiers {
		if err := modifier.Modify(req); err != nil {
			return nil, err
		}
	}

	resp, err := t.transport.RoundTrip(req)
	if err != nil {
		return nil, err
	}

	log.Debugf("%d | %s %s", resp.StatusCode, req.Method, req.URL.String())

	return resp, err
}

tokenauthorkizer.go

此文件中定义了1 个接口 3 个结构体。接口名为tokenGenerator是 token 服务的生成器。定义的三个结构体分别为：UserAgentModifier，tokenAuthorizer ，rawTokenGenerator，standardTokenGenerator 后面二个结构体都实现了tokenGenerator接口的方法用来生成不同格式的token。

tokenGenerator接口的定义如下。

1
2
3
4

type tokenGenerator interface {
	// scopes 定义了此资源运行的操作
	generate(scopes []*token.ResourceActions, endpoint string) (*models.Token, error)
}

实现了上述接口的standardTokenGenerator的generate方法从 token 服务器获取token。获取 token’的逻辑如下 1. 通过endpoint的参数调用ping()方法获得realm（领域也就是请求发送的地址） 和service，这二个参数是在认证服务器返回的Challenge参数中提取出来的。 2. 调用getToken方法，先构造好需要client 的发送请求。在请求中的查询参数有service和scope。如果credential参数不为空，需要调用Modify(req)给请求加上授权信息。完成上述工作之后就可以发送请求了。 如果请求发送成功，就可以获得token信息

实现了上述接口的rawTokenGenerator的generate方法直接调用core 文件夹下的token_util.MakeToken方法直接生成 token。

结构体tokenAuthorizer是 token 授权器的重点，其实现了registry.Modifier接口。主要负责解析出请求中scopes参数也就是需要执行的操作（pull/push），同时生成授权用的 token通过修改请求将授权用的 token 加入其中。详细定义如下：

type tokenAuthorizer struct {
	registryURL  *url.URL // used to filter request
	generator    tokenGenerator
	client       *http.Client
	cachedTokens map[string]*models.Token
	sync.Mutex
}

这里定义的请求修改器只修改发向registry的请求,下面详细介绍一下请求修改器的实现。 1. 调用filterReq方法，用来对请求进行过滤，只有发送到registry 的请求才可以通过 2. 调用parseScopes方法从请求的method, path and query string中提取出scopoes，也就是对资源的操作。 3. 如果login 操作或者scopes中只有一个参数时，从本地的cached中获取token 4. 如果经过 3 之后token还是为空，通过上述定义的二个token生成器来获取token信息，并且将这个token的信息作为 value存储在本地缓冲中，key 为scopes 5. 构建发送请求req.Header.Add(http.CanonicalHeaderKey("Authorization"), fmt.Sprintf("Bearer %s", token.Token))，为其增加token授权信息。

func NewStandardTokenAuthorizer(client *http.Client, credential Credential,
	customizedTokenService ...string) modifier.Modifier {
	generator := &standardTokenGenerator{
		credential: credential,
		client:     client,
	}

	// when the registry client is used inside Harbor, the token request
	// can be posted to token service directly rather than going through nginx.
	// If realm is set as the internal url of token service, this can resolve
	// two problems:
	// 1. performance issue
	// 2. the realm field returned by registry is an IP which can not reachable
	// inside Harbor
	if len(customizedTokenService) > 0 {
		generator.realm = customizedTokenService[0]
	}

	return &tokenAuthorizer{
		cachedTokens: make(map[string]*models.Token),
		generator:    generator,
		client:       client,
	}
}

func NewRawTokenAuthorizer(username, service string) modifier.Modifier {
	generator := &rawTokenGenerator{
		service:  service,
		username: username,
	}

	return &tokenAuthorizer{
		cachedTokens: make(map[string]*models.Token),
		generator:    generator,
	}
}

通过这二个对外暴露的方法，完美的实现了对发往registry请求的授权封装

解释一下Challenge-Response协议:基于挑战/应答（Challenge/Response）方式的身份认证系统就是每次认证时认证服务器端都给客户端发送一个不同的”挑战”字串，客户端程序收到这个”挑战”字串后，做出相应的”应答”,以此机制而研制的系统.
